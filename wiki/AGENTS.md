# AGENTS.md — 위키 운영 스키마

이 문서는 LLM 에이전트(Codex/Claude/Cursor 등)가 본 위키를 일관되게 만들고, 갱신하고, 질의에 답하기 위한 규칙입니다. 사람이 직접 위키를 쓰지 않고, LLM이 이 규칙을 따라 씁니다.

원본 패턴은 루트의 `llm-wiki.md` / `llm-wiki.ko.md` 참고.

---

## 1. 도메인 한 줄

본 프로젝트의 지식 베이스는 **국내 건강기능식품 성분·가격 비교 플랫폼(코드명 Super-Calc)** MVP를 둘러싼 시장·고객·제품·기술 정보를 누적·정제합니다.

엘리베이터 피치: *"수동 엑셀 계산과 뒷광고 필터링에 지친 건기식 소비자들을 위한, 리얼-타임 1일 단가 계산 및 의학 팩트체크 플랫폼."*

## 2. 디렉터리 구조

```
hff-hub-workbase/
├── llm-wiki.md            # 패턴 원본(EN)
├── llm-wiki.ko.md         # 패턴 원본(KO)
├── raw/                   # 원본 소스(불변, LLM이 읽기만 함)
│   ├── 00_PRD_v1.md
│   ├── 05_SRS_v1.md
│   ├── 06_value-proposition-sheet_260410(fin).md
│   ├── 1_porters-foreces.md ... 10_jtbd-interview-report.md
│   └── assets/            # 첨부 이미지 등
└── wiki/                  # LLM이 만들고 유지
    ├── AGENTS.md          # 본 파일(스키마/규약)
    ├── README.md          # 위키 진입점/개요
    ├── index.md           # 전체 페이지 카탈로그(콘텐츠 중심)
    ├── log.md             # 시간 순 로그(append-only)
    ├── synthesis.md       # 통합 종합/핵심 논제
    ├── sources/           # 원본 1건당 요약 페이지 1장
    ├── entities/          # 단일 대상 페이지(제품·세그먼트 등)
    ├── personas/          # 페르소나(C1, C2, A2, E1, E2, N1)
    ├── features/          # MVP 기능(F1~F4)
    ├── competitors/       # 경쟁사·대체재
    └── concepts/          # 분석 프레임/지표/방법론 등
```

규칙:
- `raw/` 는 절대 수정·삭제 금지. 새 원본은 같은 폴더에 추가만 한다.
- `wiki/` 안의 모든 파일은 LLM이 갱신할 수 있고, 사람은 주로 읽는 쪽이다.
- 폴더 안에서 하위 폴더를 만들기 전에 인덱스에 분류 카테고리를 먼저 정의한다.

## 3. 페이지 작성 관례

### 3.1 파일명

- 영소문자, 단어 사이는 `-` (kebab-case).
- 페르소나는 `personas/<id>-<roman-name>.md` (예: `c1-han-jeonghoon.md`).
- 기능은 `features/f<번호>-<slug>.md` (예: `f1-super-calc-engine.md`).
- 소스 요약은 원본 파일명을 단순 슬러그로 변환 (예: `sources/01-porters-five-forces.md`).

### 3.2 프론트매터(YAML)

모든 페이지는 다음 프론트매터로 시작한다.

```yaml
---
title: <한국어 제목>
type: source | entity | persona | feature | competitor | concept | meta
status: draft | stable | stale
last_updated: YYYY-MM-DD
sources: [raw/<file>.md, ...]   # 근거가 된 원본
related: [wiki/<path>.md, ...]  # 위키 내부 교차 참조
tags: [#건기식, #MVP, ...]
---
```

`status` 의미:
- `draft`: 첫 초안. 사실관계 검토 미완.
- `stable`: 사실관계 확인됨. 다른 페이지에서 인용 가능.
- `stale`: 더 새로운 원본/결정에 의해 일부가 낡음. lint 대상.

### 3.3 본문 구조 가이드

페이지 종류별 권장 섹션(엄격 강제 아님, 일관성을 위한 가이드).

- **source 페이지**
  - `## 한 줄 요약`
  - `## 핵심 포인트` (불릿)
  - `## 위키에 미친 영향` (어떤 entity/concept 페이지를 만들었거나 갱신했는가)
  - `## 인용용 발췌`
  - `## 출처`: 원본 경로 + 작성/갱신일
- **persona 페이지**
  - `## 한 줄 요약` (역할/엔진)
  - `## 프로필` (나이/직무/세그먼트/규모)
  - `## 핵심 Pain` / `## Goal` / `## 감정 곡선`
  - `## JTBD Job Story`
  - `## AOS / DOS`
  - `## 관련 기능·CJM 단계`
- **feature 페이지(F1~F4)**
  - `## 한 줄 정의`
  - `## 해결 Pain` (CORE/ADJ/EXT/CJM ID)
  - `## 핵심 AC(수용 기준)`
  - `## NFR / 임계치`
  - `## 외부·내부 의존성`
  - `## 리스크`
  - `## 관련 페르소나`
- **concept 페이지**
  - `## 정의`
  - `## 본 프로젝트에서 의미`
  - `## 핵심 도표/수치`
  - `## 결론·시사점`
- **competitor 페이지**
  - `## 한 줄 요약`
  - `## 규모·BM`
  - `## 우리와의 차별점`
  - `## 위협/기회`

### 3.4 링크·인용

- 위키 내부 링크는 항상 상대경로 `../path/to.md` 또는 `[text](path)` 형식.
- 원본 인용은 `raw/<file>.md` 경로 + (가능하면) 섹션·표 이름.
- 수치는 항상 출처와 검증 수준(공개 조사/추정)을 같이 표기한다.
- 같은 사실을 여러 페이지에서 반복할 때는 한 곳을 정본으로 정하고 다른 곳은 링크로 가리킨다.

### 3.5 ID 컨벤션

원본에서 쓰던 다음 식별자를 위키 전역에서 그대로 사용한다.

- 페르소나: `C1, C2, A2, E1, E2, N1`
- 세그먼트: `Q1, Q2, Q3, Q4` (+ `Q1-A`, `Q4-A`, `Q4-C` 등)
- Pain: `CORE-1..5, ADJ-1..3, EXT-1..4, NON-1..2, CJM-1..5`
- 기능: `F1, F2, F3, F4` (+ Should/Could `F7..F12`)
- 리스크: `R1..R5`
- 가정·검증: `A1.., V1..`, `ASM-1..`, `CON-1..`
- KPI: `North Star = TTC` 외 PRD §1-3 표 참조

## 4. 운영 워크플로

### 4.1 Ingest (새 소스 받기)

1. 새 파일을 `raw/`에 넣는다(이미지가 있으면 `raw/assets/`에).
2. LLM에게 "ingest <파일명>"이라고 지시한다.
3. LLM은 다음을 수행한다:
   1. 원본을 읽고 핵심 요약을 사용자와 짧게 확인.
   2. `wiki/sources/<slug>.md` 요약 페이지 생성·갱신.
   3. 영향받는 `entities/personas/features/concepts/competitors` 페이지를 갱신(없으면 생성).
   4. `index.md` 카탈로그 갱신.
   5. `log.md`에 한 줄 추가: `## [YYYY-MM-DD] ingest | <원본 제목> — <짧은 영향 요약>`.
4. 한 소스는 보통 5~15개 페이지를 건드린다. 너무 많으면 분할.

### 4.2 Query (위키에 질문)

1. LLM은 먼저 `index.md` → 관련 카테고리 → 후보 페이지를 좁힌다.
2. 후보 페이지를 읽어 답을 합성하고, **위키 내부 경로로 인용**한다(예: `wiki/personas/c1-han-jeonghoon.md`).
3. 답이 새 통찰(비교, 표, 다이어그램)을 만들었다면, `entities/` 또는 `concepts/`에 새 페이지로 **다시 위키에 적재**한다.
4. `log.md`에 `## [YYYY-MM-DD] query | <질문 요약> — <어디에 정리했는가>` 추가.

### 4.3 Lint (건강 검사)

주기적으로(또는 사용자가 "lint"라고 부르면) 다음을 검사한다.

- 페이지 간 모순(같은 수치가 여러 페이지에서 다르게 나옴).
- 새 원본으로 낡아진(`stale`) 주장.
- 들어오는 링크가 0인 고아 페이지.
- 본문에서 자주 언급되지만 자체 페이지가 없는 개념(예: "성분 정규화", "에셋 라이트").
- `index.md`에서 누락된 페이지.
- 검증 수준 표기가 빠진 수치.

결과는 `log.md`에 `## [YYYY-MM-DD] lint | <발견 N건> — <조치 요약>` 으로 남기고, 후속 수정은 별도 ingest/query처럼 처리.

## 5. 인덱스·로그 규약

- `index.md` 는 카테고리별로 다음 형식의 표를 유지한다.
  ```
  | 페이지 | 한 줄 요약 | 상태 | 마지막 갱신 |
  ```
- `log.md` 한 줄 시작은 항상 `## [YYYY-MM-DD] <op> | <한 줄>` 형식.
  - `op ∈ {ingest, query, lint, decision, refactor}`
  - 다음 명령으로 최신 5건 확인 가능: `grep "^## \[" wiki/log.md | tail -5`

---

## 6. 금지·주의 사항

- 원본을 wiki 페이지에 통째로 복사하지 않는다. 요약·재구성·교차 참조가 위키의 가치다.
- 마케팅·홍보 표현 금지. 위키는 내부 의사결정용 사실 베이스다.
- 건강기능식품법상 금지 표현(질병 치료/예방 등)을 위키 본문에 그대로 옮기지 않는다(원본 인용 시 따옴표·출처 명시).
- 추정 수치는 반드시 검증 수준(`✅ 공개 조사 / ⚠️ 추정 / 🔴 미검증`)을 표기한다.
- 새 페이지는 항상 `index.md`와 적어도 한 개의 다른 페이지에서 링크되어야 한다(고아 금지).

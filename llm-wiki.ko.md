# LLM 위키

LLM을 활용해 개인 지식 베이스를 만드는 패턴입니다.

이 파일은 아이디어 메모입니다. 그대로 복사해 자신의 LLM 에이전트(예: OpenAI Codex, Claude Code, OpenCode / Pi 등)에 붙여 넣도록 설계되어 있습니다. 목표는 상위 수준의 아이디어를 전달하는 것이고, 구체적인 설계는 에이전트와 함께 다듬어 나가면 됩니다.

## 핵심 아이디어

대부분의 사람이 LLM과 문서를 쓰는 방식은 RAG에 가깝습니다: 파일 묶음을 올리고, 질의 시점에 관련 청크를 검색해 답을 생성합니다. 잘 동작하지만, LLM이 매 질문마다 지식을 처음부터 다시 조립합니다. **누적이 없습니다.** 다섯 개 문서를 종합해야 하는 미묘한 질문을 하면, 매번 관련 조각을 찾아 이어 붙여야 합니다. NotebookLM, ChatGPT 파일 업로드, 대부분의 RAG 시스템이 이런 식입니다.

여기서의 아이디어는 다릅니다. 질의 시점에 원문에서만 검색하는 대신, LLM이 **지속적으로 유지·갱신하는 위키** — 당신과 원본 사이에 두는, 구조화되고 서로 링크된 마크다운 파일 모음 — 를 **점진적으로 만듭니다.** 새 소스를 추가하면, LLM이 단순히 나중에 검색하도록 색인만 두는 것이 아니라, 읽고 핵심을 뽑아 기존 위키에 **통합**합니다 — 엔티티 페이지를 갱신하고, 주제 요약을 수정하고, 새 데이터가 옛 주장과 충돌하는 곳을 표시하고, 변하는 종합 정리를 보강하거나 도전합니다. 지식은 한 번 **컴파일**되고 나서 **최신 상태로 유지**되지, 매 질문마다 다시 유도되지 않습니다.

이것이 핵심 차이입니다: **위키는 지속적이고 복리처럼 쌓이는 산출물입니다.** 교차 참조는 이미 있습니다. 모순은 이미 표시되어 있습니다. 종합은 이미 읽은 모든 것을 반영합니다. 위키는 소스를 더할 때마다, 질문을 던질 때마다 더 풍부해집니다.

위키를 직접 쓰는 일은 거의 없습니다 — LLM이 전부 씁니다. 당신은 자료 수집, 탐색, 올바른 질문에 집중합니다. LLM은 요약, 교차 참조, 분류, 장부 맞추기 같은 힘든 일을 합니다. 실제로 저는 한쪽에 LLM 에이전트, 다른 쪽에 Obsidian을 띄워 둡니다. LLM이 대화를 바탕으로 수정하고, 저는 링크를 따라가고 그래프 뷰를 보고 갱신된 페이지를 읽습니다. Obsidian이 IDE이고, LLM이 프로그래머이고, 위키가 코드베이스입니다.

맥락은 다양합니다. 예를 들면:

- **개인**: 목표, 건강, 심리, 자기계발 — 일기, 글, 팟캐스트 노트를 정리하며 시간이 지날수록 구조화된 자기 그림을 쌓기.
- **연구**: 수 주~수 개월 한 주제에 깊이 들어가기 — 논문, 글, 보고서를 읽으며 변해 가는 논제와 함께 포괄적인 위키를 쌓기.
- **책 읽기**: 읽는 대로 챕터를 넣고, 인물·주제·줄거리 실타래를 페이지로 만들며 연결하기. 끝나면 풍부한 동반 위키가 생깁니다. [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 같은 팬 위키 — 수년에 걸쳐 수천 페이지가 엮인 것 — 을 개인적으로, LLM이 교차 참조와 유지를 맡으며 비슷하게 만든다고 생각하면 됩니다.
- **비즈니스/팀**: Slack 스레드, 회의 녹초, 프로젝트 문서, 고객 통화를 먹이로 LLM이 유지하는 내부 위키. 갱신은 사람이 검토하는 방식도 가능합니다. 팀에서 아무도 하기 싫어하는 유지 작업을 LLM이 해 주므로 위키가 최신에 가깝게 유지됩니다.
- **경쟁 분석, 실사, 여행 계획, 강의 노트, 취미 깊이 파기** — 시간이 지나며 지식을 모으고 흩어지지 않게 정리하고 싶을 때.

## 구조

세 층이 있습니다.

**원본(raw sources)** — 직접 모은 출처 문서입니다. 기사, 논문, 이미지, 데이터 파일. **불변**이며 LLM은 읽기만 하고 수정하지 않습니다. 진실의 근거입니다.

**위키** — LLM이 만든 마크다운 파일들이 모인 디렉터리입니다. 요약, 엔티티 페이지, 개념 페이지, 비교, 개요, 종합. 이 층은 LLM이 전적으로 소유합니다. 페이지를 만들고, 새 소스가 오면 갱신하고, 교차 참조를 유지하며 일관성을 맞춥니다. 당신은 읽고, LLM은 씁니다.

**스키마** — 위키 구조, 관례, 소스를 받아들일 때·질문에 답할 때·위키를 유지할 때 따라야 할 워크플로를 LLM에게 알려 주는 문서입니다(예: Claude Code용 CLAUDE.md, Codex용 AGENTS.md). **핵심 설정 파일**입니다. 일반 챗봇이 아니라 규칙적인 위키 관리자로 만드는 부분입니다. 도메인에 맞게 당신과 LLM이 함께 시간이 지나며 다듬습니다.

## 운영

**수집(ingest).** 새 소스를 원본 모음에 넣고 LLM에게 처리하라고 합니다. 예: 원본을 읽고, 핵심을 당신과 짧게 논의하고, 위키에 요약 페이지를 쓰고, 색인을 갱신하고, 관련 엔티티·개념 페이지들을 고치고, 로그에 한 줄 남깁니다. 소스 하나가 위키 페이지 10~15개를 건드릴 수 있습니다. 저는 소스를 한 번에 하나씩 넣고 참여하는 쪽을 선호합니다 — 요약을 읽고, 갱신을 확인하고, 무엇을 강조할지 LLM을 이끕니다. 한꺼번에 많이 넣는 배치 수집도 덜 감독하고 가능합니다. 스타일에 맞는 워크플로를 정하고 스키마에 문서화해 이후 세션에 넘기면 됩니다.

**질의(query).** 위키를 대상으로 질문합니다. LLM이 관련 페이지를 찾아 읽고 인용과 함께 답을 합성합니다. 질문에 따라 답 형태는 다를 수 있습니다 — 마크다운 페이지, 비교 표, 슬라이드(Marp), 차트(matplotlib), 캔버스. 중요한 통찰: **좋은 답은 위키의 새 페이지로 다시 넣을 수 있습니다.** 비교 분석, 탐색에서 나온 연결 — 채팅 기록에 사라지게 두지 말고 지식 베이스에 복리처럼 쌓이게 합니다.

**점검(lint).** 주기적으로 위키 건강 검사를 시킵니다. 페이지 간 모순, 더 새 소스로 대체된 낡은 주장, 들어오는 링크가 없는 고아 페이지, 언급만 되고 페이지가 없는 중요 개념, 빠진 교차 참조, 웹 검색으로 메울 수 있는 공백 등. LLM은 조사할 새 질문과 찾을 새 소스를 잘 제안합니다. 위키가 커져도 건강을 유지하는 데 도움이 됩니다.

## 색인과 로그

위키가 커질 때 LLM(과 당신)이 길을 잃지 않게 돕는 특수 파일 두 가지입니다. 역할이 다릅니다.

**index.md**는 **내용 중심**입니다. 위키 안의 모든 것의 목록 — 링크, 한 줄 요약, 선택적으로 날짜·소스 개수 같은 메타데이터. 엔티티·개념·소스 등 카테고리별 정리. 수집할 때마다 LLM이 갱신합니다. 질의 시 LLM은 먼저 색인을 읽고 관련 페이지를 찾은 뒤 깊이 들어갑니다. 중간 규모(소스 ~100개, 페이지 수백)에서는 의외로 잘 동작하며 임베딩 RAG 인프라가 없어도 됩니다.

**log.md**는 **시간 순**입니다. 무슨 일이 언제 있었는지 — 수집, 질의, 점검 — 를 덧붙이기만 하는 기록입니다. 팁: 각 항목이 일정한 접두어로 시작하면(예: `## [2026-04-02] ingest | 기사 제목`) 단순 유닉스 도구로 파싱하기 쉽습니다 — `grep "^## \[" log.md | tail -5` 로 최근 5개 항목을 볼 수 있습니다. 로그는 위키의 연대기이자, LLM이 최근에 무엇을 했는지 파악하는 데 도움이 됩니다.

## 선택: CLI 도구

어느 시점에서는 LLM이 위키를 더 효율적으로 다루게 돕는 작은 도구를 만들고 싶어질 수 있습니다. 가장 먼저 떠오르는 것은 위키 페이지 검색입니다. 규모가 작으면 색인 파일만으로도 충분하지만, 커지면 제대로 된 검색이 필요합니다. [qmd](https://github.com/tobi/qmd)는 좋은 선택입니다: 하이브리드 BM25/벡터 검색과 LLM 재순위를 온디바이스로 하는 로컬 마크다운 검색입니다. LLM이 셸로 호출할 CLI와 MCP 서버가 모두 있습니다. 더 단순한 것을 직접 만들 수도 있고, 필요해지면 LLM과 함께 간단한 검색 스크립트를 만들면 됩니다.

## 팁

- **Obsidian Web Clipper**는 브라우저 확장으로 웹 기사를 마크다운으로 바꿉니다. 원본 모음에 빨리 넣을 때 유용합니다.
- **이미지는 로컬로 받기.** Obsidian 설정 → 파일과 링크에서 "첨부 파일 폴더 경로"를 고정 디렉터리(예: `raw/assets/`)로 둡니다. 설정 → 단축키에서 "다운로드"로 검색해 "현재 파일의 첨부 파일 다운로드"에 단축키(예: Ctrl+Shift+D)를 지정합니다. 클리핑 후 단축키를 누르면 이미지가 로컬에 저장됩니다. 선택이지만 LLM이 URL에 의존하지 않고 이미지를 직접 볼 수 있게 합니다. LLM이 인라인 이미지가 있는 마크다운을 한 번에 읽는 것은 제한이 있어서, 먼저 텍스트를 읽고 필요하면 이미지를 따로 보는 식의 우회가 필요합니다. 다소 번거롭지만 실용적입니다.
- **Obsidian 그래프 뷰**는 위키의 모양 — 무엇과 무엇이 연결되는지, 어떤 페이지가 허브인지, 고아인지 — 를 보기에 좋습니다.
- **Marp**는 마크다운 기반 슬라이드 형식입니다. Obsidian 플러그인이 있습니다. 위키 내용에서 발표 자료를 바로 만들 때 유용합니다.
- **Dataview**는 Obsidian 플러그인으로 페이지 프론트매터에 쿼리를 실행합니다. LLM이 YAML 프론트매터(태그, 날짜, 소스 개수 등)를 넣으면 Dataview로 동적 표·목록을 만들 수 있습니다.
- 위키는 마크다운 파일들의 git 저장소일 뿐입니다. 버전 이력, 브랜치, 협업을 그대로 쓸 수 있습니다.

## 왜 통하는가

지식 베이스를 유지하기 귀찮은 것은 읽거나 생각하는 일이 아니라 **장부 맞추기**입니다. 교차 참조 갱신, 요약 최신화, 새 데이터가 옛 주장과 충돌할 때 표시, 수십 페이지 일관성 유지. 사람은 유지 부담이 가치보다 빨리 커지면 위키를 버립니다. LLM은 지루해하지 않고, 교차 참조를 빼먹지 않으며, 한 번에 15개 파일을 고칠 수 있습니다. 유지 비용이 거의 없으니 위키가 유지됩니다.

사람의 역할은 자료를 선별하고, 분석 방향을 잡고, 좋은 질문을 하고, 전체가 무엇을 의미하는지 생각하는 것입니다. LLM의 역할은 그 외 모든 것입니다.

정신적으로는 Vannevar Bush의 Memex(1945) — 문서 간 연결 추적이 있는 개인적이고 선별된 지식 저장소 — 와 맞닿아 있습니다. Bush의 비전은 웹이 된 것보다 이 쪽에 가깝습니다: 비공개이고 적극적으로 선별되며, 문서 간 연겉이 문서 자체만큼 가치 있습니다. 그가 못 푼 부분은 **누가 유지할 것인가**였습니다. LLM이 그 부분을 맡습니다.

## 참고

이 문서는 의도적으로 추상적입니다. 아이디어만 설명하고 구체 구현은 다루지 않습니다. 디렉터리 구조, 스키마 관례, 페이지 형식, 도구 선택은 도메인·취향·쓰는 LLM에 따라 달라집니다. 위에 나온 것은 모두 선택·모듈형입니다 — 쓸 만한 것만 쓰고 나머지는 무시해도 됩니다. 예: 소스가 텍스트뿐이면 이미지 처리는 필요 없을 수 있습니다. 위키가 작아 색인 파일만으로 충분하면 검색 엔진도 필요 없을 수 있습니다. 슬라이드는 필요 없고 마크다운만 원할 수도 있고, 출력 형식은 전혀 다른 구성일 수도 있습니다. 사용법은 이 문서를 LLM 에이전트와 함께 공유하고, 필요에 맞게 구체화해 나가면 됩니다. 이 문서의 역할은 패턴을 전달하는 것뿐이고, 나머지는 LLM과 함께 채우면 됩니다.
